# ToyBox 模块运行策略与托管存储设计

本文档用于后续实现「混合模块架构」：保留当前模块平台，不把所有功能塞回主站；通过模块分级、平台托管存储、容器资源限制、懒启动，降低服务器长期负载。

## 1. 是否需要现在预留

建议现在预留并优先实现前两层能力：

1. `runtime.mode` 模块运行策略字段。
2. 纯前端模块和平台托管存储模块。
3. 容器模块资源限制。
4. 懒启动容器模块可以第二阶段再做。

原因：

- 后续模块仓库会依赖 `module.yaml` 契约，越早固定越不容易返工。
- 很多个人工具其实不需要独立后端容器，例如随机选择器、番茄钟、Todo、日记、收藏夹。
- `4C/24G` 服务器可以跑一些容器，但不适合无限堆常驻模块。
- 平台托管存储可以让大量轻量模块几乎零后端成本运行。

## 2. 模块分级

### 2.1 static

纯前端模块。

适合：

- 随机选择器
- 计算器
- 不需要持久化数据的工具

部署方式：

- 只构建 `frontend/dist`
- 静态产物复制到 `/module-assets/{module_id}/`
- 不创建后端容器
- 不创建模块数据库

### 2.2 platform_storage

纯前端 + 主站托管存储。

适合：

- Todo
- 日记
- 收藏夹
- 简单设置
- 轻量个人数据记录

部署方式：

- 不启动模块后端容器
- 模块前端通过 `postMessage` 请求宿主
- 宿主调用主站 API
- 主站按 `user_id + module_id + key` 隔离数据

### 2.3 container

常驻后端容器模块。

适合：

- 需要服务端密钥
- 需要复杂计算
- 需要第三方 API 服务端调用
- 需要独立数据库迁移
- 需要后台任务

部署方式：

- 维持当前 DockerRunner 模式
- 一个模块一个容器
- 加默认资源限制

### 2.4 lazy_container

懒启动后端容器模块。

适合：

- 偶尔才用
- 需要后端
- 冷启动可以接受几秒等待

部署方式：

- 安装时构建镜像、创建数据库、保存容器定义
- 平时容器停止
- 用户打开时唤醒
- 空闲超过 `idle_timeout` 后自动停止

## 3. module.yaml 契约

新增 `runtime` 字段。

```yaml
runtime:
  mode: platform_storage
  resources:
    memory: 256m
    cpus: 0.5
    pids: 128
  idle_timeout: 900
```

字段说明：

| 字段 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `runtime.mode` | enum | 自动推断 | `static` / `platform_storage` / `container` / `lazy_container` |
| `runtime.resources.memory` | string | `256m` | 容器内存上限，仅容器模式生效 |
| `runtime.resources.cpus` | number | `0.5` | 容器 CPU 上限，仅容器模式生效 |
| `runtime.resources.pids` | int | `128` | 容器进程数上限，仅容器模式生效 |
| `runtime.idle_timeout` | int | `900` | 懒启动模块空闲停止秒数 |

兼容旧模块：

- 未声明 `runtime.mode` 且 `backend.enabled: true`：按 `container` 处理。
- 未声明 `runtime.mode` 且 `backend.enabled: false`：按 `static` 处理。

示例：纯静态模块。

```yaml
runtime:
  mode: static

backend:
  enabled: false

auth:
  required: false
```

示例：平台托管存储模块。

```yaml
runtime:
  mode: platform_storage

backend:
  enabled: false

auth:
  required: true
```

示例：常驻容器模块。

```yaml
runtime:
  mode: container
  resources:
    memory: 512m
    cpus: 1.0
    pids: 256

backend:
  enabled: true
  dockerfile: backend/Dockerfile
  internal_port: 8000
  health_path: /health
```

示例：懒启动容器模块。

```yaml
runtime:
  mode: lazy_container
  resources:
    memory: 512m
    cpus: 0.5
    pids: 256
  idle_timeout: 1800

backend:
  enabled: true
  dockerfile: backend/Dockerfile
  internal_port: 8000
  health_path: /health
```

## 4. 平台托管存储

### 4.1 数据表

新增 `module_storage` 表。

```text
module_storage
  id              UUID primary key
  user_id         UUID, index, not null
  module_id       string, index, not null
  key             string, not null
  value_json      JSON, not null
  created_at      datetime
  updated_at      datetime

unique(user_id, module_id, key)
```

建议限制：

- `key` 只允许：`^[A-Za-z0-9_.:-]{1,120}$`
- 单个 value 序列化后最大 `128KB`
- 每个用户每个模块最多 `100` 个 key
- 每个用户每个模块总量默认 `2MB`
- value 只按 JSON 存储，不提供 HTML 渲染

### 4.2 后端 API

建议放在主站 `core_api.py`，只允许登录用户访问。

```http
GET /api/core/modules/{module_id}/storage/{key}
PUT /api/core/modules/{module_id}/storage/{key}
DELETE /api/core/modules/{module_id}/storage/{key}
GET /api/core/modules/{module_id}/storage?prefix=&limit=
```

PUT 请求体：

```json
{
  "value": {}
}
```

响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "key": "todos",
    "value": [],
    "updated_at": "2026-06-15T12:00:00"
  }
}
```

后端必须校验：

- 当前用户来自 `get_current_user`，不能信任前端传来的 `user_id`。
- 模块存在、未隐藏、状态为 `active`。
- 模块允许使用平台存储：`runtime.mode == "platform_storage"` 或后续显式字段 `storage.enabled: true`。
- key 合法。
- value 大小和 quota 合法。

### 4.3 模块前端 RPC

模块 iframe 不能直接 fetch 主站。沿用现有 `postMessage` 宿主代理。

模块模板新增：

```ts
export const storage = {
  get<T = unknown>(key: string, fallback?: T): Promise<T>,
  set(key: string, value: unknown): Promise<void>,
  remove(key: string): Promise<void>,
  list(prefix?: string): Promise<{ key: string; value: unknown; updated_at: string }[]>,
};
```

iframe 发给宿主的消息：

```ts
parent.postMessage({
  source: "pt-module",
  type: "storage",
  reqId,
  op: "get",
  key: "todos"
}, "*");
```

宿主返回：

```ts
iframe.contentWindow.postMessage({
  source: "pt-host",
  type: "storage_result",
  reqId,
  body: { code: 0, message: "ok", data: ... }
}, "*");
```

宿主必须校验：

- `e.source === iframe.contentWindow`
- `msg.source === "pt-module"`
- `msg.type === "storage"`
- key 合法
- body 大小合理
- module_id 只能使用当前承载页的模块，不能由 iframe 指定

## 5. 用户信息接口

容器模块后端已通过签名上下文拿到：

```text
sub
uid
uid_display
username
nickname
display_name
avatar_url
roles
module_id
ver
exp
```

对于 `static` / `platform_storage` 这类无后端模块，如果前端需要展示当前用户，可在宿主 RPC 增加只读 profile：

```ts
export function getCurrentUser(): Promise<{
  uid: number;
  uid_display: string;
  username: string;
  nickname: string;
  display_name: string;
  avatar_url: string;
  roles: string[];
}>;
```

注意：

- 这个 profile 只用于前端展示。
- 不能作为模块业务权限依据。
- 邮箱默认不提供。

## 6. 容器资源限制

当前 DockerRunner 的 `docker run` 应增加默认限制。

建议默认：

```bash
--memory 256m
--memory-swap 256m
--cpus 0.5
--pids-limit 128
--cap-drop ALL
--security-opt no-new-privileges:true
```

如果模块需要写临时文件，可加：

```bash
--tmpfs /tmp:rw,noexec,nosuid,size=64m
```

可选增强：

```bash
--read-only
```

但 `--read-only` 可能影响某些 Python 库或临时缓存，建议第二阶段开启，并给 `/tmp` tmpfs。

资源读取规则：

1. 读取 `runtime.resources`。
2. 没声明则用主站默认配置。
3. 对资源值做白名单校验。
4. 不允许模块声明超过主站上限。

建议主站配置：

```text
TOYBOX_MODULE_DEFAULT_MEMORY=256m
TOYBOX_MODULE_MAX_MEMORY=1024m
TOYBOX_MODULE_DEFAULT_CPUS=0.5
TOYBOX_MODULE_MAX_CPUS=2
TOYBOX_MODULE_DEFAULT_PIDS=128
TOYBOX_MODULE_MAX_PIDS=512
```

## 7. 懒启动设计

### 7.1 数据状态

建议给 `installed_modules` 增加：

```text
runtime_mode             string
runtime_state            string  # stopped | starting | running | failed
runtime_last_accessed_at datetime
runtime_idle_timeout     int
```

也可以第一阶段只存 manifest，第二阶段再加列。

### 7.2 安装流程

`lazy_container` 安装时：

1. clone
2. 校验 manifest
3. 构建前端
4. build image
5. 建模块库
6. 跑迁移
7. `docker create` 容器，但不常驻启动
8. 注册模块：`runtime_state=stopped`

### 7.3 访问流程

用户打开模块后，常规请求进入网关：

1. 网关发现 `runtime.mode == lazy_container`。
2. 如果容器正在运行：正常转发。
3. 如果容器停止：创建 `start_module` 任务，返回：

```json
{
  "code": 20004,
  "message": "模块正在启动，请稍后",
  "data": { "retry_after": 2 }
}
```

4. 前端显示“正在唤醒模块”，每 2 秒重试。
5. deploy_worker 执行 `docker start` + 健康检查。
6. 健康通过后更新 `runtime_state=running`。

重要原则：

- 生产环境 backend 不直接操作 Docker。
- gateway 只入队启动任务。
- deploy_worker 负责 Docker 操作。

### 7.4 空闲停止

worker 每分钟扫描：

- `runtime.mode == lazy_container`
- `runtime_state == running`
- `now - runtime_last_accessed_at > idle_timeout`

满足则：

1. `docker stop`
2. 更新 `runtime_state=stopped`

网关每次成功转发后更新 `runtime_last_accessed_at`。

## 10. 实现顺序

### 阶段 1：低风险基础

1. 增加 `runtime.mode` 校验和默认推断。
2. 支持 `static` 模块安装。
3. 增加 `module_storage` 表和 storage API。
4. 模块宿主和模板增加 storage RPC。
5. 保证现有 welcome 模块不受影响。

### 阶段 2：容器资源限制

1. 解析 `runtime.resources`。
2. DockerRunner 增加 `--memory/--cpus/--pids-limit`。
3. Admin 系统状态展示模块资源策略。

### 阶段 3：懒启动

1. 增加 runtime_state。
2. 增加 start_module 任务。
3. gateway 返回 `20004`。
4. ModuleHostPage 自动重试。
5. worker 自动停止空闲容器。

### 阶段 4：构建压力优化

可选：

- 模块仓库用 GitHub Actions 构建镜像。
- ToyBox 服务器只 `docker pull`。
- 降低 A1 服务器 build 时 CPU/内存尖峰。

## 11. 验收清单

### static 模块

- 安装一个 `backend.enabled=false/runtime.mode=static` 模块。
- 工具大厅显示。
- 点击能打开 iframe。
- 没有创建模块容器。

### platform_storage 模块

- 用户 A 写入 `todos`。
- 用户 A 刷新后能读回。
- 用户 B 读不到用户 A 的数据。
- 模块 X 读不到模块 Y 的数据。
- 超过 key/value/quota 限制会被拒绝。

### container 模块

- welcome 模块仍能安装和运行。
- 模块后端能拿到签名上下文。
- `ctx.uid_display`、`ctx.display_name`、`ctx.avatar_url` 可用。
- 改密码/重置 2FA 后旧模块 token 失效。
- `docker inspect` 能看到资源限制。

### lazy_container 模块

- 安装后容器默认 stopped。
- 首次访问返回“正在启动”状态。
- 健康检查通过后自动进入模块。
- 空闲超过 idle_timeout 后容器停止。
- 再次访问可重新唤醒。

### 安全回归

- iframe 仍是 `sandbox="allow-scripts"`。
- `/module-assets/` 仍有 `CSP: sandbox allow-scripts` 和 `Access-Control-Allow-Origin: *`。
- `/api/modules/*` 仍有 `CSP: sandbox`。
- gateway allow_paths 仍 fail-closed。
- backend 仍不挂 docker.sock。
- socket-proxy 仍只在 deploy_net。

## 12. 对当前服务器的建议

Oracle A1 免费实例 `4C/24G` 可以跑 ToyBox，但建议控制策略如下：

- static 模块：基本不限。
- platform_storage 模块：几十个通常没问题。
- container 常驻模块：先控制在 5-10 个轻量模块。
- lazy_container：适合不常用但需要后端的模块。
- 重型构建尽量转移到 GitHub Actions。

不要把所有模块集成进主站。主站只保留平台能力：账号、后台、模块注册、网关、托管存储、部署管理。具体功能优先做成 `static` 或 `platform_storage`，只有确实需要服务端能力时才使用容器。
