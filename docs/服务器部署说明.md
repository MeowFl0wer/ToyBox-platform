# ToyBox 服务器部署说明

本文档用于把 ToyBox 从 GitHub 拉取到服务器并正式部署到公网。当前建议部署版本：`v1.1.5`。

部署目标：

- 服务器只对公网开放 `80/443` 和必要的 SSH 端口。
- 主站通过 Docker Compose 运行：PostgreSQL、后端、部署 Worker、Nginx 前端、Docker socket 代理。
- 后端不直接持有 `docker.sock`；只有 `deploy_worker` 通过 `socket-proxy` 安装和运行模块。
- 生产环境必须启用 HTTPS，并使用真实邮件服务发送验证码。

## 1. 部署前准备

你需要提前准备这些信息。

| 项目 | 示例 | 填写位置 |
| --- | --- | --- |
| 域名 | `toybox.example.com` | DNS、HTTPS 反代配置、`TOYBOX_CORS_ORIGINS` |
| GitHub 平台仓库 | `https://github.com/你的账号/ToyBox-platform.git` | `git clone` 命令 |
| 部署版本 | `v1.1.5` | `git checkout v1.1.5` |
| Resend API Key | `re_xxxxx` | `deploy/.env` 的 `TOYBOX_RESEND_API_KEY` |
| Resend 发件地址 | `ToyBox <noreply@example.com>` | `deploy/.env` 的 `TOYBOX_MAIL_FROM` |
| 模块仓库 owner 白名单 | `["你的GitHub用户名"]` | `deploy/.env` 的 `TOYBOX_MODULE_SOURCE_ALLOWLIST` |
| 初始管理员用户名 | `admin` 或自定义 | `deploy/.env` 的 `TOYBOX_ADMIN_USERNAME` |
| 初始管理员邮箱 | `you@example.com` | `deploy/.env` 的 `TOYBOX_ADMIN_EMAIL` |
| 初始管理员强密码 | 至少 16 位随机密码 | `deploy/.env` 的 `TOYBOX_ADMIN_PASSWORD` |

建议服务器配置：

- Ubuntu 22.04/24.04 或 Debian 12。
- Docker Engine + Docker Compose 插件。
- 至少开放 `80/443`，SSH 端口只允许你的常用 IP 更好。
- 4 核 24GB 内存可以运行主站和少量容器模块；后续模块优先使用 `static` 或 `platform_storage`。

## 2. DNS 与公网入口

在域名服务商处添加 DNS：

```text
A  toybox.example.com  你的服务器公网 IPv4
```

如果使用 Cloudflare，请先确认代理模式和 HTTPS 模式正确。ToyBox 生产模式下 refresh cookie 是 `Secure`，所以公网入口必须是 HTTPS。只用 HTTP 会导致登录态无法稳定保持。

## 3. 安装基础环境

先进入服务器：

```bash
ssh ubuntu@你的服务器IP
```

安装 Git：

```bash
sudo apt update
sudo apt install -y git curl ca-certificates
```

安装 Docker Engine 和 Docker Compose 插件。建议按 Docker 官方文档安装，不要用很旧的系统源版本：

- Docker Ubuntu: https://docs.docker.com/engine/install/ubuntu/
- Docker Debian: https://docs.docker.com/engine/install/debian/

安装后验证：

```bash
docker --version
docker compose version
sudo systemctl enable --now docker
```

如果你希望当前用户直接运行 Docker：

```bash
sudo usermod -aG docker "$USER"
```

执行后重新登录 SSH，再验证：

```bash
docker ps
```

## 4. 从 GitHub 拉取代码

推荐部署到 `/opt/toybox`。

```bash
sudo mkdir -p /opt/toybox
sudo chown "$USER:$USER" /opt/toybox
cd /opt/toybox
git clone https://github.com/你的账号/ToyBox-platform.git ToyBox
cd ToyBox
git fetch --tags
git checkout v1.1.5
```

如果仓库是私有仓库，使用 GitHub Deploy Key 或 Personal Access Token 拉取。不要把 token 写进仓库文件。

确认版本：

```bash
git log --oneline --decorate -1
```

应该看到 `v1.1.5`。

## 5. 填写生产环境变量

复制模板：

```bash
cd /opt/toybox/ToyBox/deploy
cp .env.example .env
chmod 600 .env
```

生成强随机密钥：

```bash
openssl rand -base64 48
openssl rand -base64 48
openssl rand -base64 32
```

编辑：

```bash
nano .env
```

必须填写这些字段。

```dotenv
# PostgreSQL 超管密码：只在 Docker 内网使用，也必须足够强
POSTGRES_PASSWORD=替换成强随机密码

# JWT 签名密钥：用 openssl rand -base64 48 生成
TOYBOX_SECRET_KEY=替换成强随机密钥

# 模块用户上下文签名主密钥：用另一条 openssl rand -base64 48 生成
TOYBOX_MODULE_SIGN_KEY=替换成另一条强随机密钥

# 初始管理员。首次启动时创建，后续改 .env 不会自动改已有用户
TOYBOX_ADMIN_USERNAME=admin
TOYBOX_ADMIN_EMAIL=你的邮箱@example.com
TOYBOX_ADMIN_PASSWORD=替换成强随机管理员密码

# 生产模式：必须 false。false 后 /api/docs 会关闭，Cookie 使用生产安全配置
TOYBOX_DEV_MODE=false

# Resend 邮件。API Key 在 Resend 控制台创建
TOYBOX_RESEND_API_KEY=re_xxxxxxxx

# 发件地址必须属于 Resend 已验证域名
TOYBOX_MAIL_FROM=ToyBox <noreply@你的域名.com>

# 前端公网域名，必须是 JSON 数组字符串
TOYBOX_CORS_ORIGINS=["https://toybox.example.com"]

# 只允许安装这些 GitHub owner 的模块仓库。强烈建议生产开启
TOYBOX_MODULE_SOURCE_ALLOWLIST=["你的GitHub用户名或组织名"]
```

Resend 需要做两件事：

1. 在 Resend 控制台创建 API Key，填入 `TOYBOX_RESEND_API_KEY`。
2. 在 Resend 验证你的发件域名，然后把 `TOYBOX_MAIL_FROM` 设置成该域名下的地址，例如 `ToyBox <noreply@example.com>`。

不要把 `deploy/.env` 提交到 GitHub。项目已通过 `.gitignore` 忽略 `.env`，但服务器上也要注意权限。

## 6. HTTPS 推荐方案

推荐用外层 HTTPS 反代终止 TLS，然后把流量转给 ToyBox 的 web 容器。

这样做的好处：

- 证书申请和续期交给外层反代，ToyBox 容器只监听本机端口。
- 可以避免把未加密的 `80:80` 直接作为公网入口。
- 后续换域名或加 WAF/CDN 更容易。

### 方案 A：Caddy 反代（推荐）

安装 Caddy 可参考官方文档：https://caddyserver.com/docs/install

先把 `deploy/docker-compose.yml` 里的 web 端口改成只监听本机。找到：

```yaml
ports:
  - "80:80"
```

改成：

```yaml
ports:
  - "127.0.0.1:8080:80"
```

这样 Caddy 才能独占公网 `80/443` 申请和续期证书。不要让 Caddy 和 ToyBox web 容器同时绑定公网 `80`，否则其中一个会启动失败。

然后配置 Caddy：

```bash
sudo nano /etc/caddy/Caddyfile
```

写入：

```caddyfile
toybox.example.com {
    encode zstd gzip
    reverse_proxy 127.0.0.1:8080

    header {
        Strict-Transport-Security "max-age=31536000; includeSubDomains"
        X-Content-Type-Options "nosniff"
        Referrer-Policy "no-referrer"
        X-Frame-Options "SAMEORIGIN"
    }
}
```

重载：

```bash
sudo systemctl reload caddy
```

Caddy 会自动申请和续期证书。确认服务器防火墙开放 `80/443`。

### 方案 B：使用已有宝塔、1Panel、Nginx Proxy Manager、Cloudflare Tunnel

把 ToyBox web 容器改成只监听本机：

```yaml
ports:
  - "127.0.0.1:8080:80"
```

不要保留原来的 `"80:80"`，否则外层面板和 ToyBox web 容器会抢公网 `80`。

然后在外层面板里添加 HTTPS 站点：

```text
域名：toybox.example.com
反代目标：http://127.0.0.1:8080
WebSocket：不需要
强制 HTTPS：开启
HSTS：建议开启
```

注意：如果外层反代会传 `X-Forwarded-For`，`deploy/docker-compose.yml` 已给 backend 设置 `TOYBOX_TRUST_PROXY=true`。外层反代必须只允许可信来源访问后端入口，不要让用户伪造 `X-Forwarded-For` 直连 backend。当前 compose 没有暴露 backend 端口，保持这样。

### 方案 C：直接使用容器内 Nginx 的 443

也可以使用 `deploy/nginx.conf` 里的 443 模板，但你需要：

1. 把证书挂进 web 容器。
2. 在 `deploy/docker-compose.yml` 开放 `443:443`。
3. 修改 `deploy/nginx.conf` 里的 `ssl_certificate` 和 `ssl_certificate_key`。
4. 把 `80` server 改成跳转 `443`。

这个方案能用，但证书管理会更麻烦。首次部署更推荐方案 A 或 B。

## 8. 首次登录和管理员安全

浏览器打开：

```text
https://toybox.example.com
```

使用 `.env` 中的管理员账号登录。

首次管理员登录会要求绑定 Authenticator 动态验证码：

1. 用 Google Authenticator、Microsoft Authenticator、1Password、Bitwarden 等 App 添加账户。
2. 页面会显示 Setup Key，手动输入到 App。
3. 输入 6 位动态码完成绑定。
4. 保存一次性恢复码。恢复码只显示一次，丢了只能通过服务器 CLI 或另一个管理员重置。

登录后建议立刻做：

- 修改管理员密码。
- 确认管理员邮箱正确。
- 进入后台管理，检查系统状态、用户管理、访问统计、模块管理。

如果管理员 2FA 丢失，可 SSH 到服务器后执行：

```bash
cd /opt/toybox/ToyBox/deploy
docker compose exec backend python -m app.manage list-admins
docker compose exec backend python -m app.manage reset-totp admin
```

## 9. 模块部署说明

ToyBox 支持四种模块运行模式：

| 模式 | 是否需要容器 | 适合场景 |
| --- | --- | --- |
| `static` | 不需要 | 纯展示、纯前端、小工具 |
| `platform_storage` | 不需要 | 前端工具 + 少量用户数据，使用主站托管存储 |
| `container` | 需要 | 有独立后端、数据库、复杂计算 |
| `lazy_container` | 当前仍按常驻容器运行 | 预留懒启动模式，真正按需启动是后续阶段 |

正式服务器上建议：

- 优先做 `static` 或 `platform_storage`，成本最低。
- 少量必要功能再用 `container`。
- 每个容器模块都要在 `module.yaml` 写 `gateway.allow_paths`，只开放业务接口。
- 不要随便使用 `gateway.legacy_allow_all: true`。
- `TOYBOX_MODULE_SOURCE_ALLOWLIST` 必须限制成你自己的 GitHub owner。

模块安装入口：

1. 登录管理员。
2. 进入后台管理。
3. 打开模块部署。
4. 填 GitHub 模块仓库地址和分支/标签。
5. 等待安装任务完成。

如果模块构建失败，看：

```bash
docker compose logs -f deploy_worker
```

v1.1.3 中 `socket-proxy` 已开启 `BUILD: 1`，`deploy_worker` 已设置 `DOCKER_BUILDKIT=0`，服务器本机构建模块镜像应能通过代理。

## 10. 安全加固清单

上线前逐项确认。

### 10.1 账号与密钥

- `TOYBOX_DEV_MODE=false`。
- `TOYBOX_ADMIN_PASSWORD` 不是默认值，且足够强。
- `TOYBOX_SECRET_KEY` 和 `TOYBOX_MODULE_SIGN_KEY` 是两条不同的强随机值。
- `POSTGRES_PASSWORD` 是强随机值。
- 管理员已绑定 Authenticator。
- 恢复码已离线保存。
- `deploy/.env` 权限为 `600`。

### 10.2 网络暴露面

公网只开放：

```text
22/tcp  SSH，最好限制来源 IP
80/tcp  HTTPS 证书申请和跳转
443/tcp HTTPS 正式访问
```

不要开放：

```text
5432/tcp PostgreSQL
8000/tcp backend
2375/tcp Docker API / socket-proxy
```

检查端口：

```bash
sudo ss -lntp
```

如果用 UFW：

```bash
sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
sudo ufw status
```

如果是 Oracle Cloud，也要在云控制台的安全列表/网络安全组中只放行 `22/80/443`。

### 10.3 HTTPS 与 Cookie

- 公网访问必须是 `https://`。
- `http://` 应跳转到 `https://`。
- 浏览器开发者工具里确认 refresh cookie 是 `HttpOnly`、`Secure`。
- 如果登录后刷新就掉线，优先检查是否真的走 HTTPS，以及 `TOYBOX_CORS_ORIGINS` 是否填成你的 HTTPS 域名。

### 10.4 模块隔离

当前系统已有这些保护：

- 模块前端在 sandbox iframe 中运行。
- `/module-assets/` 响应有 `Content-Security-Policy: sandbox allow-scripts`。
- 模块 API 网关有路径白名单，未声明 `gateway.allow_paths` 会 fail-closed。
- 后端模块通过短期模块 token 调用网关，不带主站 Cookie。
- 模块用户上下文使用每模块派生密钥签名。
- 容器模块运行时有限制：内存、CPU、PID、cap-drop、no-new-privileges。
- `socket-proxy` 只在 `deploy_net`，模块容器无法访问。

仍然需要你遵守：

- 只安装你自己开发或审查过的模块。
- 模块仓库 owner 白名单必须开启。
- 不要给模块配置过大的资源。
- 不要运行多个 `deploy_worker` 副本；当前任务队列按单 worker 设计。
- 后续模块多了以后，优先把构建迁到 GitHub Actions，服务器只 `docker pull + run`，然后移除 `BUILD: 1`。

### 10.5 主站 CSP

`deploy/nginx.conf` 中主站 CSP 默认注释，因为前端使用内联样式，直接开启可能误伤页面。上线稳定后可在浏览器自测通过再启用：

```nginx
add_header Content-Security-Policy "default-src 'self'; img-src 'self' data:; style-src 'self' 'unsafe-inline'; script-src 'self'; frame-src 'self'; connect-src 'self'; base-uri 'self'; form-action 'self'" always;
```

启用后必须测试：

- 首页正常。
- 登录弹窗正常。
- 后台管理正常。
- 模块 iframe 正常。
- 浏览器控制台没有 CSP 报错。

## 11. 备份与恢复

ToyBox 生产数据主要在两个 Docker volume：

```text
toybox_pgdata  PostgreSQL 数据
toybox_data    模块源码、模块前端产物、头像、平台密钥等
```

创建备份目录：

```bash
mkdir -p /opt/toybox/backups
```

备份数据库：

```bash
cd /opt/toybox/ToyBox/deploy
docker compose exec -T db pg_dump -U postgres main_db > /opt/toybox/backups/main_db_$(date +%F).sql
```

备份 data volume：

```bash
docker run --rm \
  -v toybox_data:/data:ro \
  -v /opt/toybox/backups:/backup \
  alpine tar czf /backup/toybox_data_$(date +%F).tgz -C /data .
```

建议：

- 每次升级前手动备份。
- 定期把备份同步到服务器外部。
- 恢复流程先在测试机演练，不要第一次恢复就在生产机上试。

## 13. 常见问题

### 13.1 生产启动失败，提示默认密码或占位密钥

原因：`TOYBOX_DEV_MODE=false` 时，系统会在启动时拒绝以下不安全配置并直接退出：

- 管理员密码是默认 `Admin@12345`、含 `change-me`/`change-this` 占位串，或长度不足 12 位。
- `TOYBOX_SECRET_KEY` / `TOYBOX_MODULE_SIGN_KEY` 含 `change-me`/`change-this` 占位串。
- `POSTGRES_PASSWORD`（注入到数据库连接串）含 `change-me`/`change-this` 占位串。

处理：检查 `deploy/.env`，把下面这些都改成强随机值（不要保留 `.env.example` 的占位）：

```dotenv
POSTGRES_PASSWORD=强随机值（不要含 change-this）
TOYBOX_ADMIN_PASSWORD=强随机值，至少 12 位（不要用 Admin@12345 / change-this）
TOYBOX_SECRET_KEY=openssl rand -base64 48（不要含 change-me/change-this）
TOYBOX_MODULE_SIGN_KEY=另一条 openssl rand -base64 48（不要含 change-me/change-this）
```

### 13.2 登录成功后刷新就退出

优先检查：

- 是否使用 `https://` 访问。
- 外层反代是否正确转发到 ToyBox。
- `TOYBOX_CORS_ORIGINS` 是否是 `["https://你的域名"]`。
- 浏览器是否阻止 Cookie。

### 13.3 邮件验证码发不出去

检查：

```dotenv
TOYBOX_DEV_MODE=false
TOYBOX_RESEND_API_KEY=re_xxxxx
TOYBOX_MAIL_FROM=ToyBox <noreply@你的域名.com>
```

同时确认：

- Resend API Key 没填错。
- 发件域名已在 Resend 验证完成。
- 服务器能访问 `https://api.resend.com`。

查看日志：

```bash
docker compose logs -f backend
```

### 13.4 模块安装卡住

v1.1.3 已加入崩溃回收：worker 重启时会把进行中的旧任务标记为 failed。

查看：

```bash
docker compose logs -f deploy_worker
docker compose restart deploy_worker
```

注意不要启动多个 `deploy_worker` 副本。

### 13.5 模块构建报 Docker 权限或 403 / 找不到 docker 命令

若报 `[Errno 2] No such file or directory: 'docker'`（找不到 docker 命令）：说明部署镜像里没有
docker 客户端。v1.1.5 起 `deploy/backend.Dockerfile` 改用官方静态 docker CLI 并在构建时自检，
更新到 v1.1.5+ 后**重新构建镜像**即可：

```bash
cd /opt/toybox/ToyBox && git fetch --tags && git checkout v1.1.5
cd deploy && docker compose build --no-cache backend deploy_worker && docker compose up -d
```

若报 Docker 权限或 403：确认 `deploy/docker-compose.yml` 中：

```yaml
socket-proxy:
  environment:
    BUILD: 1
    POST: 1
    EXEC: 0

deploy_worker:
  environment:
    DOCKER_BUILDKIT: "0"
```

然后重启：

```bash
docker compose up -d --build
```

### 13.6 欢迎小屋或模块 iframe 空白

检查响应头：

```bash
curl -I https://toybox.example.com/module-assets/welcome/index.html
```

应包含：

```text
Content-Security-Policy: sandbox allow-scripts
Access-Control-Allow-Origin: *
```

如果没有，检查 `deploy/nginx.conf` 是否被外层反代覆盖，或者是否访问到了错误服务。
