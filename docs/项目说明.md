# ToyBox 个人工具平台 · 详细说明

> 一个部署在个人服务器上的私有工具平台：主站负责登录/用户/工具大厅/后台管理/模块网关，
> 各功能以**独立模块**形式从 GitHub 安装、独立运行；新增功能不改主站核心代码。

- 三个仓库
  - **ToyBox**（纯前端模板，保留备份）：https://github.com/MeowFl0wer/ToyBox
  - **ToyBox-platform**（实现版，前后端 + 部署）：https://github.com/MeowFl0wer/ToyBox-platform
  - **personal-tool-module-welcome**（功能模块模板 / 示例）：https://github.com/MeowFl0wer/personal-tool-module-welcome

---

## 一、网站概述

ToyBox 是「主站平台 + 独立模块」架构：

- **主站（稳定壳层）**：登录注册、用户体系、工具大厅、模块安装/部署、模块网关、后台管理、网站内容、运行管理。主站长期稳定，不承载具体业务功能。
- **模块（独立应用）**：每个功能是一个独立 GitHub 仓库，有自己的前端、后端、（可选）数据库与迁移；通过统一模板开发，经后台一键安装、构建、部署、上线，出现在工具大厅，用户点击后在 iframe 中使用。
- **统一登录**：用户只在主站登录一次；模块通过主站网关下发的**签名用户上下文**识别当前用户，并按 `user_id` 隔离数据。

技术栈：

| 层 | 技术 |
| --- | --- |
| 主站前端 | React 18 + TypeScript + Vite + Tailwind + Motion（沿用原 ToyBox 视觉） |
| 主站后端 | FastAPI + SQLAlchemy 2 + Pydantic（SQLite 开发 / PostgreSQL 生产） |
| 部署 | Docker Compose + Nginx + PostgreSQL；模块容器加入内部网络 |
| 邮件 | Resend（开发态降级为返回验证码） |

## 二、实现的功能

### 2.1 账号与用户
- 注册（**邮箱验证码**）、登录、退出、刷新登录态、查看/修改资料（昵称、简介、头像）。
- **找回密码 / 修改邮箱 / 修改密码**：均需邮箱验证码（改邮箱验证码发往新邮箱、改密码需旧密码 + 验证码）。
- 用户字段：`uid`（从 1 递增、展示为 `000001`、被禁用/注销也**绝不复用**）、内部 `id`(UUID)、`username`(唯一)、`nickname`、`avatar`、`bio`、`email`(唯一)、`role`(user/admin)、`status`。

### 2.2 工具大厅
- 模块列表来自**后端注册表**（非前端写死）；展示图标、名称、简介、分类、登录要求与状态。
- 支持**搜索**、**分类筛选**、**收藏**（收藏分类 + 卡片右上角星标）、最近使用计数。
- 点击模块 → 统一模块承载页（iframe）；即将上线/需登录有相应提示。

### 2.3 后台管理（仅管理员）
分页布局：概览 / 模块管理 / **模块部署（GitHub 一键安装）** / 内容编辑 / 用户管理 / 系统状态 / 访问统计。
- 模块管理：列表、隐藏/取消隐藏、卸载（内置不可卸载）。
- 模块部署：填仓库地址安装，实时显示任务状态与日志。
- 内容编辑：站点文案编辑 + 发布「即将上线」入口。
- 用户管理：列表、禁用/启用。
- 模块管理还支持**重启**与**查看日志**（restart/logs）。
- **系统状态**：后端/数据库/部署方式、CPU/内存/磁盘占用、各模块健康。
- **访问统计**：今日/总 PV·UV、近 7 天趋势、访问路径 Top、模块访问 Top。

### 2.4 模块平台
- **一键安装/部署**：clone → 校验 `module.yaml` → 构建前端 → 部署后端 → 健康检查 → 注册上线。
- **Module Gateway**：`/api/modules/{id}/*` 鉴权 + 签名用户上下文转发。
- **两套部署方式**：`local`（开发，进程+SQLite）/ `docker`（生产，镜像+容器+模块独立 Postgres 库）。
- 主站重启自动重新拉起 active 模块；本地方式用 pidfile 清理，杜绝端口泄漏。

### 2.5 安全加固
bcrypt 密码哈希；JWT 短期 access + HttpOnly Cookie 刷新令牌（轮换）；登录/注册/验证码**限流**；
Pydantic 强校验 + 文本清洗；富文本/Markdown 用 bleach 白名单 + 纯文本去标签**防 XSS**；ORM 参数化**防注入**；
安全响应头；CORS 白名单；管理操作**审计日志**；模块来源限制（仅 https GitHub）；模块只信任主站 HMAC 签名上下文。

## 三、逻辑（架构）

```
浏览器
  │
Nginx ── /                → 主站前端（SPA）
  │   ── /module-assets/  → 模块前端静态资源（iframe）
  │   ── /api/*           → 主站后端 FastAPI
  │
主站后端
  ├─ Auth / Users         登录注册、用户、邮箱验证码
  ├─ Core API             工具大厅、收藏、站点内容
  ├─ Admin API            模块/用户/内容/审计
  ├─ Module Installer     安装流水线（后台线程）
  ├─ Module Gateway       /api/modules/{id}/* 鉴权 + 签名转发
  └─ Runner（local/docker）部署运行模块
        │
   模块后端容器（内部网络，不暴露公网）
        └─ 模块独立 PostgreSQL database
```

核心原则：
- **主站稳定**：主站数据库与核心代码不为模块业务改动。
- **模块独立**：模块不能访问主站库或其它模块库；只能通过主站签名上下文识别用户。
- **统一登录**：模块声明 `auth.required`；需登录模块未登录不可进入，登录态由网关注入。

代码位置（`server/app/`）：
`api/auth.py`（认证）、`api/core_api.py`（工具大厅/内容）、`api/admin.py`（后台）、`api/gateway.py`（网关）、
`modules_runtime.py`（安装流水线 + Local/Docker Runner）、`models.py`（数据模型）、`core/security.py`（密码/JWT/签名/清洗）。

## 四、数据

主站库 `main_db` 只存平台级数据（`server/app/models.py`）：

| 表 | 用途 |
| --- | --- |
| `users` | 用户（含 `uid` 展示号、role、status） |
| `sequences` | 单调计数器，分配 `uid`（保证不复用） |
| `user_sessions` | 刷新令牌会话（存哈希、可吊销） |
| `email_verifications` | 邮箱验证码（哈希存储、限次限时、一次性、按用途区分） |
| `installed_modules` | 已安装模块注册表（含 `internal_backend_url`、`manifest`、状态、隐藏） |
| `module_install_jobs` | 安装/部署任务（状态、日志） |
| `module_user_preferences` | 用户对模块的收藏/置顶/最近使用 |
| `site_contents` | 后台可编辑的站点文案 |
| `admin_audit_logs` | 管理操作审计 |

模块业务数据**不进主站库**：每个需要库的模块使用独立数据库（生产 `module_<id>_db`，开发 SQLite 文件），
业务表必须含 `user_id` 并在查询/写入时强制隔离。

## 五、运行链路

### 5.1 登录态恢复（打开页面）
前端挂载 → `POST /api/auth/refresh`（带 HttpOnly Cookie）→ 拿到新 access token（仅存内存）→ `GET /api/auth/me` 还原用户。

### 5.2 注册（邮箱验证码）
`POST /api/auth/register/send-code`（生成验证码，Resend 发信；开发态直接返回）→
`POST /api/auth/register`（校验码 + 唯一性 + 分配 uid + 建号 + 下发 token/Cookie）。
找回密码 / 改邮箱 / 改密码同理：先 send-code 再带码提交。

### 5.3 模块安装（后台 → 上线）
`POST /api/admin/modules/install`（建任务，后台线程执行）：
```
clone 仓库 → 读 module.yaml 校验（id/版本/路径） → 构建前端(产物 → modules/assets/<id>)
  → Runner 部署后端：
       local  : 建模块 venv 装依赖 →（如需）SQLite + alembic → uvicorn 子进程
       docker : docker build 镜像 → 建模块独立 Postgres 库/用户 →（如需）容器内 alembic
                 → docker run 加入内部网络（不暴露端口）
  → /health 健康检查 → 写 installed_modules、状态 active → 上线
```
前端轮询 `GET /api/admin/modules/install-jobs/{job_id}` 看状态与日志。

### 5.4 使用模块（网关请求 · iframe 隔离）
用户进入 `工具大厅 → 模块` → 模块承载页加载 **sandbox(allow-scripts) 的 iframe**
（不透明源，拿不到主站 Cookie、不能访问 `/api/auth/refresh`）→ 宿主向后端取**模块级短期 token**
（`POST /api/core/modules/{id}/token`，需登录）→ 模块前端**不直接联网**，而是经 `postMessage`
把请求交给宿主，宿主用模块 token 调 `/api/modules/{id}/<path>` →
**主站网关**：检查模块已安装/启用/运行 → 用**模块 token**鉴权（不再凭 Cookie）→
生成签名上下文：
```
X-PT-Module-Id, X-PT-User-Context=base64url(json), X-PT-User-Signature=HMAC_SHA256(MODULE_SIGN_KEY, 上下文)
```
→ 转发到模块后端 `internal_backend_url` → 模块校验签名、取 `user.sub` 读写自己的库 → 透传响应。

## 六、生产部署

### 6.1 主站部署（Docker Compose）

前置：一台装好 Docker / Docker Compose 的服务器；域名解析到服务器。

```bash
git clone https://github.com/MeowFl0wer/ToyBox-platform
cd ToyBox-platform/deploy
cp .env.example .env
# 编辑 .env：POSTGRES_PASSWORD、TOYBOX_SECRET_KEY、TOYBOX_MODULE_SIGN_KEY、
#            管理员账号、Resend(TOYBOX_RESEND_API_KEY/MAIL_FROM)、TOYBOX_CORS_ORIGINS
docker compose up -d --build
```

起来后：
- 主站：`http://服务器/`（建议前置 TLS：在 web 服务加 443 + 证书，或用外层反代）。
- 健康检查：`/api/health`；接口文档：`/api/docs`。
- 默认管理员：`.env` 里设置的账号，**首次登录后立即改密码**。

组件（见 `deploy/docker-compose.yml`）：
- `db`（PostgreSQL，容器名 `toybox-postgres`）：主库 `main_db` + 各模块独立库。
- `backend`（FastAPI）：`TOYBOX_DEPLOY_MODE=docker`；挂载 `docker.sock` 以构建/运行模块容器；
  与 `web` 共享 `data` 卷交付模块前端产物。
- `web`（Nginx）：托管主站前端 + `/module-assets`，反代 `/api`。
- 网络 `toybox_net`：模块容器加入，不暴露公网端口。

安全要点（已按最小权限设计）：
- **主站 `backend` 不持有 `docker.sock`**。安装/构建/运行/卸载等需要 Docker 的操作交给独立的
  `deploy_worker`，它经 **`docker-socket-proxy`** 访问**受限的** Docker API（仅 containers/images/networks，
  禁用 `exec`）；建模块独立库由 worker 用 **psycopg 直连 Postgres** 完成，连 EXEC 都无需开放。
- 所有密钥用强随机值；`TOYBOX_MODULE_SIGN_KEY` 主站与模块共享，泄露会被伪造用户身份。
- 上线务必配置 HTTPS，并把刷新 Cookie 的 `secure` 设为 True（生产可在 `auth.py` 调整）。
- 进一步收敛：可改为 CI 构建模块镜像并推镜像仓库、worker 仅 `pull` 运行，则 socket-proxy 权限更小。

任务执行位置由 `TOYBOX_WORKER_INPROC` 控制：开发态 `true`（主站进程内线程，无需 Docker/worker）；
生产 `false`（主站只入队，`deploy_worker` 轮询执行）。

### 6.2 模块安装（管理员操作）

1. 登录主站 → 进入「后台管理 → 模块部署」。
2. 填模块 GitHub 仓库地址（仅支持 `https://github.com/...`），可填分支/Tag/Commit。
3. 点「安装并部署」，观察任务状态与日志直到「已上线」。
4. 模块出现在工具大厅，点击即可使用。
5. 「模块管理」里可隐藏/取消隐藏、卸载（卸载会停容器并删资源）。

示例（用模板模块验证整条链路）：仓库填
`https://github.com/MeowFl0wer/personal-tool-module-welcome` → 安装后工具大厅出现「欢迎小屋」，登录进入显示「欢迎 xxx！！」。

### 6.3 模块模板使用

模块模板仓库：`personal-tool-module-welcome`（也内置于本仓库 `module-template/`）。

开发新模块：
```bash
# 复制模板为新仓库
git clone https://github.com/MeowFl0wer/personal-tool-module-welcome personal-tool-module-<your-id>
cd personal-tool-module-<your-id> && rm -rf .git && git init
```
按需修改后推到你自己的 GitHub，再在后台填地址安装。模板结构：
```
module.yaml          # 模块清单
frontend/            # 模块前端（Vite，构建到 frontend/dist，iframe 承载）
backend/             # 模块后端（FastAPI：/health、/manifest、业务接口）
  app/core/auth.py   # 校验主站签名用户上下文
scripts/             # build_frontend / build_backend / run_dev
README.md            # 接口契约与开发指引
```

### 6.4 模块开发注意事项与接口对齐

**必须实现的统一契约**：
- `module.yaml`：`id`(小写)、`version`、`entry`(iframe/frontend_dist)、`backend`(dockerfile/internal_port/health_path)、`database`(enabled)、`auth`(required)、`permissions`。
- 后端 `GET /health`（部署器健康检查）、`GET /manifest`（返回清单）。
- 所有接口返回统一响应 `{ "code": 0, "message": "ok", "data": ... }`。
- 业务接口经主站网关 `/api/modules/{id}/<path>` 访问（模块内即 `/<path>`）。

**模块前端（iframe 内）**：在 sandbox 不透明源中运行，不能直接 fetch 主站。统一用模板提供的
`frontend/src/api.ts`（postMessage RPC，由宿主代发并附模块 token），模块开发者只调 `call()/getGreeting()`；
动态文本务必用 `textContent`（模板已示范），勿用 innerHTML 注入业务数据。

**签名用户上下文（关键，务必对齐）**：每模块签名密钥独立（主站派生 `HMAC(主密钥, module_id)` 并注入模块 env）。
主站网关下发三个请求头，方案见 `server/app/core/security.py: build_module_user_headers` 与模板 `backend/app/core/auth.py`：
```
X-PT-User-Context   = base64url( json.dumps(payload, 紧凑, key 排序) )
X-PT-User-Signature = hmac_sha256_hex( MODULE_SIGN_KEY, X-PT-User-Context )
```
模块用环境变量 `MODULE_SIGN_KEY`（主站部署时注入，与主站一致）重算签名并恒定时间比对，再 base64 解码取用户：
```python
ctx = Depends(require_user)     # 需登录接口：匿名/无 sub → 401
ctx.sub / ctx.username / ctx.roles
```

**需要数据库的模块**：`module.yaml` 设 `database.enabled: true`，提供 `backend/alembic` 迁移；
主站会创建独立库并注入 `DATABASE_URL`。业务表**必须含 `user_id`**，查询/写入都用 `ctx.sub` 强制隔离。

**不需要登录的模块**：`auth.required: false` → 可匿名进入，但**不得持久化任何业务数据**（每次打开为初始状态）。

**模块不能做**：访问主站库或其它模块库；自行处理主站登录；信任前端传来的 `user_id`；暴露公网端口；修改主站文件。

## 七、本地开发

```bash
# 后端（SQLite，本地进程部署模式，无需 Docker）
cd server && bash run.sh          # http://127.0.0.1:8000

# 前端（/api、/module-assets 已代理到后端）
npm install && npm run dev        # http://localhost:5173
```
默认管理员 `admin / Admin@12345`；邮箱验证码开发态在接口返回并自动填入；模块安装走本地进程方式可直接验证。
